# 研发部门工作计划管理系统：核心需求清单 (PRD) - v5.0

## 1. 核心设计思想：三级目标契约
* **Level 1 (团队目标)：** 部门团队目标类目的定义与维护。
* **Level 1.5 (月度调度)：** 每月对团队目标类目进行“权重权重”与“资源预算”的动态分配。
* **Level 2 (执行纽带)：** 以项目Tag 全生命周期管理为核心，支持跨月滚动，记录项目全周期工时。
* **Level 3 (个人执行)：** 周承诺 (P) 与 日流水 (A) 登记，登记过程中关联项目Tag，关联影响项目进度。

---

## 2. 核心功能模块需求

### 2.1 战略规划与月度调度 (负责人端)
* **功能点 1：团队目标类目库 (Level 1)**
    * 管理长期稳定的目标（如：常规运维、产品&项目研发、团队日常）。
* **功能点 2：月度权重与配置 (Level 1.5)**
    * 每月对目标类目设置 `weight`（分值权重）和 `budget_days`（人天预算）。
    * *逻辑：* 同一个目标，3月权重 20%，4月可调至 10%。
* **功能点 3：Tag 项目全周期管理 (Level 2)**
    * 新建或承接上月未完成项目Tag，设定项目起止时间和人天预算，并关联至 L1 团队目标类目。
    * **跨月规则：** 只要 `status` 未标记为“已完成”，Tag 自动出现在下月规划池中。

### 2.2 执行与减负登记 (全员端)
* **功能点 4：周承诺录入 (Level 3)**
    * 基于活跃 Tag 录入本周承诺任务及预计工时 $P$。
* **功能点 5：工作登记（优化） (Level 3)**
    * 工作登记页面新增“项目Tag”关联。这样工作流水才能与项目完成执行关联。
* **功能点 6：日常项快捷引用 (核心优化)**
    * **引用机制：** 员工可将任一 `dev_work_log` 记录设为“日常操作”。
    * **一键登记：** 在登记页面点击该快捷入口，系统自动继承 `tag_id`、工作内容、工时 $A$，仅 `log_date` 默认为当日。
    * **真实性控制：** 必须由员工手动点击提交，不自动生成，以确保数据符合实际出勤情况。

### 2.3 监控与决策分析 (管理端)
* **功能点 7：项目进度看板**
    * 展示 Tag 的累计投入人天 ($A$)、初始预估 ($P$) 及逻辑完成度。
* **功能点 8：资源负荷参考**
    * 在制定下月计划时，自动展示留存 Tag 已占用的资源比例。
* **功能点 9：风险预警灯**
    * 当项目工时严重偏离进度或 $A$ 严重偏离 $P$ 时触发预警。

## 3. 整合后的数据表设计 (重点：整合最新内容)

基于您的原始表，我进行了必要的字段扩展，以支持最新业务逻辑：

| 表名 | 关键字段 (加粗为新增/调整) | 说明 |
| :--- | :--- | :--- |
| **`goal_define`** | `goal_id`, `name` | **Level 1：** 团队目标类目定义表（如“常规运维”） |
| **`goal_config`** | `config_id`, `goal_id` (FK),  `year`, **`month`**, `weight`, `budget_days` | **Level 1.5：** 月度权重配置表（按月存储权重） |
| `project_tag` | `tag_id`, `goal_id` (FK), `year`, `tag_name`, `start_date`, `end_date`, `budget_days`, `status` | **Level 2：** 纽带表。`status` 控制跨月滚动逻辑 |
| `user_task` | `task_id`, `user_id`, `tag_id` (FK), `year`, `month`, `week`, `p_hours`, `task_status` | **Level 3：** 周承诺计划表 |
| `dev_work_log` | `log_id`, `user_id`, `tag_id` (FK), `a_hours`, `log_date`, **`content`**, **`is_shortcut`** | **Level 3：** 实际流水表。`is_shortcut` 标记该记录是否为日常快捷项 |

---

## 4. 关键业务逻辑规则

1.  **“流水不断，任务不灭”：**
    * `project_tag` 的统计不以自然月为物理边界。查询某 Tag 的总投入时，累加该 `tag_id` 下所有日期的 `a_hours`。
    * 系统前端在每月 1 日自动拉取所有 `status = '进行中'` 的 Tag。

2.  **“日常项引用”克隆逻辑：**
    * 当员工在 `dev_work_log` 中标记某条记录为 `is_shortcut = true` 时，该记录进入快捷区。
    * 点击快捷项后：`New_Log.tag_id = Old_Log.tag_id`, `New_Log.content = Old_Log.content`, `New_Log.a_hours = Old_Log.a_hours`, `New_Log.log_date = Today`。

3.  **“考核月结”逻辑：**
    * 绩效分值 = (当月某类目下 Tag 的进度贡献) × (当月 `goal_config` 中该类目的 `weight`)。
    * 这确保了即便任务跨月，管理上的激励权重依然可以每月灵活调整。

## 5. 实施路线建议

* **第一阶段：** 升级 `dev_work_log` 增加快捷引用功能，实现“秒级登记”。
* **第二阶段：** 建立 `goal_config` 的月度映射关系，实现负责人的“资源动态调度”。
* **第三阶段：** 自动汇总跨月 Tag 进度，生成“全局设计”看板。
