# DevWorkLog 项目升级需求文档 V1

## 1. 文档目的

基于现有 `dev-work-log-server`、`dev-work-log-ui` 项目能力，以及《研发计划设计V1.md》的升级目标，形成本次项目升级的产品与技术设计文档，用于指导后续后端建模、前端页面改造、数据库升级与接口开发。

## 2. 现状分析

### 2.1 现有后端能力

当前后端主要包含以下模块：

- 用户管理：`sys_user` 用户基础信息、登录、密码修改、账号启停、分页查询。
- 工作日志：`dev_work_log` 的新增、编辑、删除、分页查询、导出、按产品/类别/日期汇总。
- 阻塞预警：`dev_blocking_alert` 根据工作日志状态为“已挂起”自动生成预警记录，并支持标记解决。
- 仪表盘：按时间范围统计总工时、业务工时占比、Bug 修复占比、阻塞任务数、工时分布、周趋势、明细查询。

### 2.2 现有前端能力

当前前端主要包含以下页面：

- 登录页
- 汇总看板
- 每日工作登记
- 工作日志汇总
- 账号管理
- 个人中心

### 2.3 现有系统主要问题

- 当前系统以“日常工作日志登记”为核心，缺少“计划 -> 执行 -> 复盘”的闭环。
- 工作日志只能关联产品与任务类别，无法关联具体研发计划或项目 Tag。
- 缺少部门级目标配置、月度资源预算、跨月项目滚动管理能力。
- 缺少个人周规划录入能力，无法形成 P（计划）与 A（实际）对比。
- 当前阻塞预警仅依赖“已挂起”状态，无法基于项目偏差、工时偏差做过程性预警。
- 仪表盘统计维度以日志维度为主，无法支撑项目进度、资源负荷、绩效归因。

## 3. 本次升级目标

将当前“研发工作登记系统”升级为“研发计划与执行管理系统”，构建三级管理模型：

- Level 1：团队目标分类定义
- Level 1.5：月度目标权重与资源预算配置
- Level 2：项目 Tag 全生命周期管理
- Level 3：个人周规划与日流水执行登记

升级后系统需要实现：

- 从“记录做了什么”升级为“围绕计划管理执行结果”
- 将每日日志与项目 Tag 强关联
- 支持跨月滚动管理未完成项目
- 支持基于 P/A 的执行偏差分析
- 支持项目进度、资源占用、风险预警、月度绩效归因

## 4. 角色定义

### 4.1 管理员/负责人

- 维护团队目标分类
- 配置月度权重与预算
- 创建/维护项目 Tag
- 查看资源负荷、项目进度、风险预警、统计报表

### 4.2 普通研发成员

- 查看自己可参与的活跃 Tag
- 录入周规划任务
- 录入每日工作日志
- 使用快捷引用登记日常重复性工作
- 查看个人计划与执行情况

## 5. 需求文档

### 5.1 一级需求：目标体系管理

#### 5.1.1 团队目标分类管理

系统应支持负责人维护团队目标分类，例如：

- 常规运维
- 产品&项目研发
- 团队日常

每个目标分类需要支持启用/停用、排序、说明维护。

#### 5.1.2 月度权重与预算配置

系统应支持负责人按“年 + 月 + 目标分类”维护：

- 权重 weight
- 预算人天 budget_days

同一目标分类在不同月份可配置不同权重与预算，以支撑资源动态调整。

### 5.2 二级需求：项目 Tag 管理

#### 5.2.1 项目 Tag 建立

系统应支持负责人创建项目 Tag，并维护：

- 关联目标分类
- Tag 名称
- 项目描述
- 起止日期
- 预算人天
- 责任人
- 参与成员
- 项目状态

#### 5.2.2 跨月滚动

当项目 Tag 未完成时，在下一个月份自动进入规划池；只有状态变为“已完成/已关闭”时才停止滚动。

#### 5.2.3 项目进度核算

系统应根据个人周规划和每日实际日志，自动汇总：

- 累计计划工时 P
- 累计实际工时 A
- 工时偏差
- 项目完成度

### 5.3 三级需求：个人执行管理

#### 5.3.1 周规划录入

员工应可按周录入承诺任务，必须关联活跃 Tag，并填写：

- 所属年月周
- 预计工时 P
- 本周任务说明
- 任务状态

#### 5.3.2 每日工作登记升级

每日工作登记页面需从原有字段：

- 关联产品
- 任务类别

升级为优先关联：

- 项目 Tag
- 工作内容
- 实际工时 A
- 日志日期
- 状态

保留产品、类别作为兼容字段或衍生字段，用于平滑迁移历史数据。

#### 5.3.3 日常工作快捷引用

员工可将某条工作日志设为“快捷引用模板”。

使用快捷引用时系统自动带出：

- tag_id
- content
- a_hours

用户只需确认当天日期后手动提交。

### 5.4 四级需求：监控与分析

#### 5.4.1 项目进度看板

系统应展示项目维度：

- 预算人天
- 累计计划工时
- 累计实际工时
- 完成率
- 偏差率
- 阻塞状态

#### 5.4.2 资源负荷分析

系统应支持按月查看：

- 各目标分类预算占比
- 各项目 Tag 实际资源投入
- 跨月项目占用资源

#### 5.4.3 风险预警

满足以下条件时生成风险预警：

- 项目实际工时显著高于计划工时
- 项目实际投入接近或超过预算人天
- Tag 长时间无更新
- Tag 状态阻塞
- 周规划长期未完成

#### 5.4.4 月度考核归因

系统应支持按月统计：

- 目标分类下项目完成贡献
- 按目标分类应用月度权重
- 形成月度绩效参考数据

## 7. 核心功能业务流程

### 7.1 月度规划流程

1. 负责人维护目标分类。
2. 负责人为某月配置目标分类权重与预算人天。
3. 负责人创建当月项目 Tag，关联目标分类并设置预算与起止时间。
4. 系统自动带出上月未完成的 Tag 进入当前月规划池。
5. 负责人确认 Tag 责任人与参与成员。

### 7.2 周规划流程

1. 员工进入“周规划”页面。
2. 系统展示自己可参与的活跃 Tag。
3. 员工选择 Tag，填写本周任务和预计工时 P。
4. 提交后生成 `user_task` 记录。
5. 管理端可按周查看个人承诺与项目承诺汇总。

### 7.3 每日执行登记流程

1. 员工进入“每日工作登记”页面。
2. 系统优先展示活跃 Tag 列表和快捷引用入口。
3. 员工选择 Tag 或快捷模板。
4. 填写/确认工作内容、实际工时 A、日志日期、状态。
5. 提交后生成 `dev_work_log` 记录。
6. 系统自动回写该 Tag 的累计实际工时与最近更新时间。

### 7.4 快捷引用流程

1. 员工在历史日志中将某条记录标记为快捷引用。
2. 系统在“每日工作登记”页面展示该快捷项。
3. 员工点击快捷引用。
4. 系统自动复制历史记录的 Tag、内容、工时，日期默认当天。
5. 员工确认后手动提交。

### 7.5 风险预警流程

1. 系统定时扫描项目 Tag、周规划、工作日志。
2. 判断是否满足预警规则，如 A > P、预算超限、长期阻塞等。
3. 生成预警记录。
4. 管理员在预警中心查看并处理。
5. 处理完成后标记为已解决或已忽略。

### 7.6 月末考核流程

1. 系统按月汇总项目实际执行数据。
2. 将项目归集到目标分类。
3. 读取当月 `goal_config` 的权重配置。
4. 生成目标分类贡献值和参考绩效统计。

- 在保留现有 `sys_user`、`dev_work_log`、`dev_blocking_alert` 的基础上扩展。

- 对跨月项目、计划工时、实际工时和预警记录进行结构化建模。

## 10. 接口设计文档

统一约定：

- 前缀：`/api/v2`
- 返回结构沿用当前风格：`{ code, msg, data }`
- 登录与用户体系可先复用 v1

### 10.1 目标分类管理

#### 10.1.1 查询目标分类列表

- `GET /api/v2/goals`

请求参数：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| keyword | string | 名称关键词，可选 |
| status | number | 状态，可选 |

返回：

```json
{
  "code": 200,
  "msg": "success",
  "data": [
    {
      "goal_id": 1,
      "goal_name": "产品研发",
      "status": 1
    }
  ]
}
```

#### 10.1.2 新增目标分类

- `POST /api/v2/goals`

请求体：

```json
{
  "goal_name": "产品研发",
  "goal_desc": "面向产品能力建设",
  "sort_no": 10,
  "status": 1
}
```

#### 10.1.3 修改目标分类

- `PUT /api/v2/goals/:goalId`

#### 10.1.4 启停目标分类

- `PUT /api/v2/goals/:goalId/status`

### 10.2 月度配置管理

#### 10.2.1 查询月度配置

- `GET /api/v2/goal-configs?year=2026&month=3`

返回：

```json
{
  "code": 200,
  "msg": "success",
  "data": [
    {
      "config_id": 1,
      "goal_id": 1,
      "goal_name": "产品研发",

      "month": 3,

      "budget_days": 42

  ]
}
```

```json
{
  "year": 2026,
  "month": 3,
  "configs": [

      "goal_id": 1,
```

请求参数：

## 11. 升级实施建议

### 11.1 数据迁移建议

现有 `dev_work_log` 历史数据没有 `tag_id`，建议采用如下方式平滑升级：

1. 先为 `dev_work_log` 新增 `tag_id`、`is_shortcut`、`source_log_id` 字段，允许为空。
2. 历史数据先保留原 `product_type`、`task_category` 逻辑继续可查询。
3. 新增日志全部要求关联 `tag_id`。
4. 对高频产品或项目，逐步人工补录映射为新的 Tag。

### 11.2 前端页面升级建议

建议新增或改造页面如下：

- 更名“汇总看板”为“工作汇总”，保留现在的“汇总看板”全部内容，作为默认Tab页，
  新增一个“日志汇总”TAB页，把当前的“工作日志”放到新增的“日志汇总”TAB页中。
- 升级“每日工作登记”为 项目Tag联动，并支持常规工作的快捷引用
- 新增“团队目标管理”包含“目标分类管理” 和 “月度资源规划” 两个Tab，默认是 “月度资源规划”
- 新增“工作规划”页，该页分为“周计划”和“项目管理”两项tab。周计划着重是个人周规划任务的管理
  项目管理则是项目管理者对项目的整体计管理。
- 新增“项目资源看板”，体现目标/项目/资源相关汇总和统计
- 新增“风险预警中心”

### 11.3 后端改造建议

- 保留 `/api/v1` 作为兼容接口
- 新增 `/api/v2` 完成计划管理能力
- 优先完成模型新增与聚合接口
- 预警规则建议拆分为服务层或定时任务

## 12. 结论

本次升级的本质不是“给日志表多加几个字段”，而是把系统从“工作记录工具”升级为“研发计划执行管理平台”。

结合现有项目代码，推荐按以下顺序推进：

1. 数据模型升级：目标分类、月度配置、项目 Tag、周规划。
2. 每日登记改造：日志强关联 项目Tag，并支持快捷引用。
3. 项目看板新增：从项目/目标/资源统计。
4. 风险预警升级：从单条日志挂起扩展到项目偏差预警。
5. 月度考核归因：引入权重和预算数据，形成绩效参考。

以上方案能够兼容现有项目结构，并且与《研发计划设计V1.md》的升级方向保持一致。
